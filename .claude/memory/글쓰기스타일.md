# 글쓰기 스타일 가이드

> 태현님의 글쓰기 스타일을 기록합니다.
> `/new-post` 실행 시 이 파일을 참고해서 AI 느낌 없이 작성합니다.

마지막 업데이트: 2026-05-27

---

## 핵심 원칙

**"완벽한 설명보다 솔직한 경험"**

- 이 블로그는 백엔드 개발자로 성장하는 과정을 기록하는 공간
- 직접 겪은 경험, 삽질, 고민 과정, 해결 과정을 솔직하게
- "왜 이런 선택을 했는가"를 중심으로

---

## ✅ 좋은 표현

| 상황 | 좋은 표현 |
|------|----------|
| 시작 | "처음 써봤는데", "이걸 왜 쓰게 됐냐면" |
| 문제 | "막혔다", "삽질했다", "이상하게 안 됐다" |
| 해결 | "알고 보니", "결국엔", "이렇게 해결했다" |
| 느낌 | "생각보다 편했다", "솔직히 기대 이하였다" |
| 마무리 | "다음엔 ~을 해봐야겠다", "나중에 다시 보면" |

---

## ❌ 피할 표현

- "본 포스트에서는 ~에 대해 알아보겠습니다"
- "~은 다음과 같은 특징을 가집니다"
- "따라서 우리는 ~을 사용할 수 있습니다"
- "이상으로 ~에 대해 살펴보았습니다"
- "독자 여러분"

---

## 글 구조

```
[도입] 이걸 왜 쓰게 됐는지 — 상황 설명 (2~3문장)

[본문] 과정 중심 서술
  - 시도한 것
  - 막힌 부분
  - 어떻게 해결했는지
  - (코드가 있다면 코드 포함)

[마무리] 배운 점 or 다음 계획 (1~2문장)
```

---

## 문체 참고 (기존 포스트 기반)

기존 포스트 "GitHub 기술 블로그를 시작한 이유" 의 스타일:
- 짧은 문장 선호
- 빈 줄로 여백 확보
- `---` 로 섹션 구분
- 개인적인 생각을 솔직하게 표현
- H1(`#`) 헤딩으로 섹션 구분
